# 需求规格说明

## 1. 项目背景

船舶水动力学仿真通常依赖 CFD、势流理论、水池试验和经验公式。传统 CFD 在精度和通用性方面较强，但前处理复杂、计算资源消耗高、迭代周期长。Physics-Informed Neural Network，简称 PINN，可以把控制方程、边界条件、初始条件、观测数据和工程约束共同纳入神经网络训练过程，为快速仿真、参数扫描、反演识别和代理模型构建提供新的技术路径。

本系统面向船舶水动力学研究、船型方案快速评估和教学实验，建设一个基于 PhysicsNeMo 的 PINN 仿真平台。系统提供船型/算例管理、物理场景配置、PINN 训练与推理、结果可视化、报告导出、数据追踪和模型版本管理能力。

## 2. 建设目标

### 2.1 总体目标

构建一个可扩展、可追踪、可复现实验的船舶水动力学 PINN 仿真系统，使用户能够通过 Web 界面完成仿真算例配置、模型训练、仿真推理、结果分析和报告生成。

### 2.2 阶段目标

- V1 原型阶段：支持二维或简化三维流场的 PINN 算例管理、训练任务提交、后台执行、基础结果可视化和报告导出。
- V2 工程化阶段：支持船体几何资产、边界条件模板、训练配置模板、模型检查点、多用户权限和数据库驱动的后台任务管理。
- V3 扩展阶段：支持更多水动力场景，例如阻力预报、规则波中运动响应、操纵性简化模型、参数扫描和数据同化。
- V4 平台增强阶段：按需引入独立任务队列、实时消息推送与分布式缓存能力，以支撑更高并发和更复杂部署。

## 4. 系统边界

### 4.1 系统包含

- 用户、角色与权限管理。
- 项目、船型、几何资产与仿真算例管理。
- 水动力物理场景配置，包括流体参数、边界条件、初始条件、控制方程和训练采样域。
- PhysicsNeMo PINN 模型配置、训练、推理和检查点管理。
- 后台任务执行、状态跟踪、日志记录和失败重试。
- 流场、压力、速度、自由液面、残差、损失曲线等结果的可视化。
- 仿真结果对比、报告导出和实验归档。
- 数据库与文件资产管理。

### 4.2 系统暂不包含

- 高保真工业级全船 RANS CFD 的完整替代。
- CAD 建模器本身的复杂几何编辑能力。
- 大规模 HPC 作业调度系统的深度集成。后续可通过适配器接入 Slurm、Kubernetes 或云 GPU 集群。
- 商业船级社规范自动判定。可作为后续插件扩展。
- 基于 Redis 的独立任务队列、实时消息推送和分布式缓存架构。当前版本不纳入，后续在并发规模和部署复杂度提升后再引入。

## 8. 典型使用场景

### UC-001 创建静水阻力仿真算例

1. 用户创建项目并录入船型参数。
2. 用户上传船体几何或选择参数化船型模板。
3. 用户选择静水阻力仿真类型。
4. 用户配置来流速度、密度、黏度、边界条件和计算域。
5. 用户选择 PINN 模型模板并设置训练参数。
6. 系统提交训练任务并显示状态、日志和损失曲线。
7. 训练完成后，用户执行推理并查看压力场、速度场和阻力系数。
8. 用户导出报告。

### UC-002 多方案对比

1. 用户复制基准算例为多个方案。
2. 用户修改船型参数、速度或边界条件。
3. 系统批量提交训练或推理任务。
4. 用户在结果对比视图中查看阻力、升力、残差、收敛曲线和关键剖面。
5. 用户生成对比报告。

### UC-003 基于观测数据的数据同化

1. 用户上传水池试验或 CFD 参考数据。
2. 用户配置观测点、物理量、单位和数据质量说明。
3. 用户在 PINN 损失项中启用观测数据损失。
4. 系统训练模型并输出物理残差与数据拟合误差。
5. 用户评估模型泛化能力。

## 9. 约束与假设

- PhysicsNeMo 作为 PINN 训练与物理约束建模的核心框架。
- Flask 负责业务 API、鉴权、任务提交和数据管理，不直接承载长时间训练。
- React 负责交互式配置、任务监控和结果可视化。
- 长耗时训练任务通过异步 Worker 执行。
- 当前版本以后端 Worker 配合 PostgreSQL 任务表为主，不将 Redis 作为前置依赖。
- 数据库保存结构化元数据，几何文件、检查点和大体积结果文件保存在文件存储中。
- 初期以研究级仿真平台为目标，不承诺替代经过验证的工业 CFD 软件。

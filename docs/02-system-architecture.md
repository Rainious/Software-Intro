# 总体软件架构设计

## 1. 架构目标

系统架构应服务于三个核心目标：

- 将 Web 业务系统与长耗时 PINN 训练解耦，保证交互体验稳定。
- 将水动力物理建模、神经网络训练、后处理和可视化分层，便于扩展新的方程、损失项和仿真场景。
- 对实验配置、数据、模型、结果进行版本化和可追踪管理，保证科研与工程分析的可复现性。

## 2. 技术栈

| 层级 | 技术选择 | 说明 |
| --- | --- | --- |
| 前端 | React + JavaScript | 构建配置界面、任务监控、结果可视化和报告页面 |
| 后端 API | Python Flask | 提供 REST API、鉴权、业务编排、任务提交和元数据管理 |
| PINN 引擎 | NVIDIA PhysicsNeMo | 定义网络、物理约束、采样域、训练流程和推理流程 |
| 数据库 | PostgreSQL | 保存用户、项目、算例、任务、配置、结果元数据 |
| 后台任务 | Python Worker（数据库驱动） | 基于 PostgreSQL 任务表执行训练、推理、后处理和报告生成 |
| 文件存储 | MinIO/S3 或本地文件系统 | 保存几何文件、数据集、检查点、图像和报告 |
| 可视化 | React 图表库 + Three.js/VTK.js 可选 | 二维图表、三维几何、流场和切片展示 |
| 部署 | Docker Compose 起步，后续 Kubernetes | 便于环境隔离和扩展 GPU Worker |

## 3. 总体分层

```text
用户浏览器
  |
  v
React 前端层
  |
  v
Flask API 与应用服务层
  |
  +--> PostgreSQL 元数据层
  +--> 对象存储/文件存储层
  +--> 异步 Worker 层
           |
           v
       PhysicsNeMo PINN 引擎层
           |
           v
       训练结果、检查点、后处理工件
```

## 4. 逻辑架构

### 4.1 表现层

表现层由 React 实现，负责：

- 用户登录、项目导航和权限感知 UI。
- 船型、几何、算例、模型配置的表单化编辑。
- 任务状态、日志、损失曲线和资源占用展示。
- 流场云图、矢量图、剖面曲线、对比图和报告预览。
- 与后端 REST API 通信，不直接访问数据库或训练引擎。

### 4.2 API 与应用服务层

Flask 后端负责：

- API 路由、鉴权、权限校验和参数校验。
- 项目、船型、算例、数据集、模型配置和结果的业务管理。
- 任务编排，创建任务记录并由 Worker 领取执行训练、推理、后处理和报告生成任务。
- 保存配置快照和状态变更。
- 生成前端所需的轻量结果索引。

### 4.3 领域层

领域层封装核心业务概念：

- Project：项目上下文与权限范围。
- Vessel：船型参数和几何引用。
- SimulationCase：仿真算例与物理场景配置。
- PhysicsScenario：方程、边界条件、初始条件和流体参数。
- PINNModelConfig：网络结构、约束、损失权重和训练参数。
- Job：训练、推理、后处理、报告等异步任务。
- Result：可视化结果、指标、报告和归档资产。

### 4.4 PINN 引擎层

PINN 引擎层基于 PhysicsNeMo 构建，负责：

- 将数据库中的算例配置转换为 PhysicsNeMo 可执行配置。
- 根据仿真类型加载控制方程、边界条件和采样策略。
- 构造神经网络、约束节点、损失函数和优化器。
- 执行训练、检查点保存、推理采样和误差评估。
- 输出标准化结果文件和指标 JSON，供后端入库和前端展示。

### 4.5 数据层

数据层由两类主存储组成：

- PostgreSQL：结构化元数据，例如用户、项目、算例、任务、模型版本和结果索引，同时承担后台任务记录与状态源。
- 对象存储：大文件，例如几何文件、训练数据、检查点、VTK/NPY/CSV、图片、报告。

## 5. 部署架构

### 5.1 开发环境

```text
本机开发环境
  ├─ frontend dev server
  ├─ Flask API
  ├─ PostgreSQL
  ├─ MinIO 或本地 storage
  └─ Worker，可使用 CPU 或单卡 GPU
```

### 5.2 生产环境

```text
负载均衡/反向代理
  ├─ React 静态资源服务
  ├─ Flask API 服务实例
  ├─ Worker
  │    ├─ CPU Worker：后处理、报告、轻量推理
  │    └─ GPU Worker：PhysicsNeMo 训练和高成本推理
  ├─ PostgreSQL 主库
  ├─ 对象存储
  └─ 日志与监控系统
```

## 6. 关键设计原则

### 6.1 训练任务异步化

训练和高成本推理不在 Flask 请求线程内执行。API 只负责校验配置、创建 Job 记录并返回任务 ID。Worker 通过数据库任务表领取待执行任务，并按任务类型加载 PhysicsNeMo 执行。

### 6.2 配置快照不可变

每次训练或推理任务启动时，系统都保存完整配置快照，包括算例参数、边界条件、网络配置、依赖版本、代码版本和随机种子。任务开始后，历史快照不随用户后续编辑而改变。

### 6.3 模型与结果版本化

同一算例可以有多个模型版本和结果版本。每个版本对应明确的训练任务、检查点、配置快照和输出指标。

### 6.4 引擎适配层隔离

Flask 业务服务不直接拼接 PhysicsNeMo 内部对象。应通过 `physicsnemo_adapter` 或类似适配层将业务配置转换为引擎配置，降低框架升级带来的影响。

### 6.5 大文件外置

数据库不存储大体积二进制结果，只保存 URI、元数据、校验值、大小和访问权限。

## 7. 数据流

### 7.1 训练数据流

1. 前端提交训练配置。
2. Flask 校验权限与配置合法性。
3. 后端创建 `jobs` 与 `training_jobs` 记录，保存配置快照。
4. GPU Worker 从数据库领取待执行任务并调用 PhysicsNeMo。
6. Worker 定期写入训练日志、损失指标和进度。
7. Worker 保存检查点与结果工件到对象存储。
8. Worker 更新任务状态为成功或失败。
9. 前端通过轮询接收状态更新。

### 7.2 推理与后处理数据流

1. 用户选择模型检查点和采样配置。
2. Flask 创建推理任务。
3. Worker 加载模型并生成流场、剖面、曲线或指标。
4. 后处理模块计算阻力系数、压力分布、残差统计等指标。
5. 结果文件进入对象存储，结果索引写入数据库。
6. 前端按需加载轻量数据或分片数据用于可视化。

## 8. 安全架构

- 所有 API 默认要求认证，公开接口需要明确标记。
- 项目数据按项目成员权限隔离。
- 上传文件需要校验类型、大小、哈希和病毒扫描策略。
- 对象存储文件通过后端签名 URL 或代理下载控制访问。
- 关键操作写入审计日志。
- 训练任务运行容器应限制文件系统权限和网络访问范围。

## 9. 可观测性

| 类型 | 内容 |
| --- | --- |
| API 日志 | 请求 ID、用户、路径、耗时、状态码 |
| 任务日志 | Job ID、阶段、进度、错误堆栈、资源信息 |
| 训练日志 | iteration、loss、各损失项、学习率、采样数 |
| 指标监控 | CPU/GPU 使用率、显存、待执行任务数、任务耗时 |
| 审计日志 | 用户操作、对象类型、对象 ID、变更摘要 |

## 10. 架构风险与缓解

| 风险 | 影响 | 缓解方式 |
| --- | --- | --- |
| PINN 对复杂三维湍流问题收敛困难 | 结果不稳定或训练成本过高 | 分阶段支持，从简化方程和基准算例开始，建立验证集 |
| PhysicsNeMo API 版本变化 | 引擎层适配成本增加 | 建立适配层和配置转换测试 |
| 大体积结果可视化卡顿 | 前端体验差 | 结果分层、降采样、切片加载和服务端预处理 |
| 多用户 GPU 资源竞争 | 任务等待时间长 | 任务优先级、资源配额、CPU/GPU Worker 分离和资源监控 |
| 实验不可复现 | 科研可信度下降 | 强制配置快照、版本记录、随机种子和依赖记录 |

## 11. 后续平台增强

- 当前版本不将 Redis 作为基础依赖。
- 当任务并发、跨节点部署或实时日志推送需求明显增长时，后续版本可再引入 Redis + Celery/RQ 或等价方案。
